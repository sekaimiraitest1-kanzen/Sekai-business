# RLS audit — 2026-04-30

## TL;DR

- **All 13 `public` tables have RLS enabled** (verified `pg_tables.rowsecurity = true`)
- **30 policies** spanning every CRUD path, gated by `is_admin_of(salon_id)` SECURITY DEFINER fn for admin operations
- **No critical findings**
- 1 medium (booking insert spam vector — by-design, mitigation deferred to V1.1)
- 2 low (multi-tenant scaffold gaps — V1 is single-tenant, not exploitable)

## Findings

### Finding 1 — Anonymous INSERT on `bookings`/`orders`/`customers` (MEDIUM, by-design)

**What**: Three policies allow anonymous INSERT with no `with_check` constraint:

```
bookings  insert guest   INSERT  (with_check: true)
orders    insert guest   INSERT  (with_check: true)
customers insert guest   INSERT  (with_check: true)
```

This is intentional — guests must be able to book/order without authentication. The booking flow at `/zakazivanje` and the shop checkout at `/shop/[slug]` both rely on this.

**Risk**: A scripted client (curl, Python) hitting the anon endpoint can spam:
- Fake bookings → calendar pollution, denial-of-service for real customers
- Fake orders → admin notification flood, possibly Resend email rate limit hit
- Fake customer rows → DB bloat (each row ~500 bytes; 100K spam rows = 50MB toward Supabase Free 500MB cap)

**Mitigation status**: NOT implemented. Acceptable for V1 because:
1. Public domain not announced yet → discovery risk near-zero pre-launch
2. Single-tenant, low-traffic launch → real-world spam unlikely first weeks
3. Admin can manually delete spam rows if it appears

**V1.1 plan** (only ship if real spam observed):
- IP-based rate limit on booking insert: max 3/hour/IP via Vercel Edge Middleware
- Honeypot field in booking form (hidden input; if filled, reject)
- Optional: hCaptcha on booking submit (free tier 1M/mo)

Tracked in `docs/release-master-plan-2026-04-30.md` post-launch section.

### Finding 2 — `salons` publicly readable (LOW, multi-tenant scaffold)

**What**: `salons.public read salons` policy has `qual: true` — every row visible to anon.

**Risk**: If multi-tenant later, competitor salons could read each other's contact info, working hours, etc.

**Status for V1**: Single-tenant — only `slug = 'trisa'` row exists. Not exploitable.

**V1.1 plan**: Tighten to `qual: (slug = current_setting('request.salon_slug', true))` or per-domain matching when multi-tenant lands. Or simply scope SELECT to `slug = ANY (active_slugs)`.

### Finding 3 — `site_content` publicly readable with `qual: true` (LOW, expected)

**What**: Same `qual: true` shape as salons.

**Risk**: All content rows readable. But content is meant to be on the site by definition — no leak.

**Status**: Acceptable. No action.

## What we verified did NOT exist

- ✅ No tables with RLS disabled (`rowsecurity = false`)
- ✅ No `cmd: ALL` policy with `qual: true` (would equal "anon can do anything")
- ✅ No `cmd: DELETE` policies for anon role
- ✅ No `cmd: ALL` admin policy missing the `is_admin_of(...)` check
- ✅ Admin paths in app code consistently use `service_role` client (which bypasses RLS) via `createAdminClient()` — verified in 23 admin files

---

## Service-role key handling

- Stored in Vercel env as `SUPABASE_SERVICE_ROLE_KEY` (Sensitive flag)
- Used only in `createAdminClient()` server-side
- Never exposed to client bundle (verified: no `NEXT_PUBLIC_` prefix)
- Rotation: NOT planned for V1 — current key is project-creation key. **Schedule rotation in V1.1** as best-practice hygiene (Supabase dashboard → Settings → API → "Reset service_role key").

---

## Auth flow

- Admin auth = PIN-based, custom (`admin_users.pin_hash` bcrypt) — does NOT use Supabase Auth
- Supabase `auth.uid()` therefore always NULL for current admin sessions
- Admin RLS policies bypass via service_role client
- Customer-side auth (when added in V1.1 for "my orders" page) WILL use Supabase Auth, at which point the `auth.uid()` based policies in customers/bookings/orders/loyalty_events activate
